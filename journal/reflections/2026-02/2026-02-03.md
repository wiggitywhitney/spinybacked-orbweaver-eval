When I think about creating an OpenTelemetry instrumenting agent, I have some initial thoughts:
- I think this should be multiple PRDs and not one monolithic PRD. There needs to be some thought about what framework to make, what to add, and in what order, generally speaking.
- There also needs to be research done around the old commit-story V1 repo. In that repo, I instrumented every single little thing, and perhaps I'll do that again with this agent. It might be easier, weirdly, than teaching it to make judgment calls, but that is a future decision.
- Specifically in the old commit-story V1, I want to research the add telemetry slash command. That's the specific thing to study.
- This new tool will definitely use the OpenTelemetry weaver schema, and I think there should be its own tool for reading and updating the schema. Maybe those are even two different tools. That would let the agent propose new attributes for gaps.
- Perhaps attributes need their own tools, so there would be tools for logs, metrics, traces, and attributes.
- Also, within each of those tools, we want to ensure that when multiple signals are added, they're correlated with one another.
Those are all the things that are on my mind. Also, in the old repo, look at the telemetry standards there.
